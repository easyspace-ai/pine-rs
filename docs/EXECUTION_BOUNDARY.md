# 执行边界与实时执行路线（pine-rs / pine-tv / playground）

> **用途**：定义 pine-rs 系列项目在当前阶段的**执行边界**、实时执行策略与后续开发顺序。  
> **适用范围**：`pine-rs` 内核、`pine-tv` 服务端、`playground` 前端。  
> **优先级**：本文件补充 [`AGENTS.md`](../AGENTS.md) 的阶段目标；若与工程实现冲突，以“不要虚报已完成、不要降低门禁”为先。

> **说明**：下文关于“TradingView VM 执行机制”的内容，表示**本项目对齐目标和实现抽象**。  
> 我们参考的是官方公开文档中的执行模型、数据订阅方式与产品行为；**不把这些文字写成对 TradingView 内部私有实现的绝对断言**。

## 1. 当前决策

### 1.1 总原则

当前阶段不再把“前端 JavaScript 计算指标”当成长期方案。  
下一阶段的目标是：**尽可能统一到 pine-rs / WASM / 后端会话执行链路**，减少两套语义长期并存。

换句话说：

- **前端 JS 指标实现**只视为**当前过渡能力**，不继续扩张覆盖面。
- **正式能力边界**优先向“统一 Pine 执行模型”收敛。
- **实时能力建设**以后端脚本会话和统一行情时间线为主线推进。

### 1.2 本轮明确取舍

| 区域 | 处理原则 |
|---|---|
| 前端轻指标（MA / EMA / RSI / MACD / BOLL） | **暂不继续投入**；保留现状，后续由 WASM 统一替换 |
| Pine 脚本 / 自定义脚本 | **后端主执行** |
| 策略脚本 | **后端主执行** |
| 告警前置条件 / 信号计算 | **后端主执行** |
| 实时细粒度语义 | 先列缺口，不在本轮宣称已对齐 |

## 4. 目标执行模型（本项目要实现的机制）

### 4.1 三层核心机制

#### 机制 A：历史 bar 顺序执行

目标行为：

- 脚本从第一根历史 bar 顺序执行到最后一根
- 每根 bar 执行一次
- 每次执行后的结果写入时间序列

这是所有 Pine 语义的基础。如果历史循环不稳定，后面的实时行为不会可信。

#### 机制 B：实时 bar 执行 + rollback

目标行为：

- 当实时 bar 收到新 tick 时，先恢复到“上一根已收盘 bar 提交后的状态”
- 再用当前 forming bar 的最新值重跑脚本
- 只有 bar close 时，才把当前状态正式提交

这决定了实时图上最后一根 bar 的正确性，也是避免实时结果累计漂移的核心。

#### 机制 C：`varip` 不参与 rollback

目标行为：

- `regular`：每次执行重新初始化
- `var`：跨 bar 持久，但实时 bar 内每次 tick 回滚
- `varip`：跨 tick 保留，不参与 rollback

这关系到 intrabar 统计、逐 tick 状态累计和大量脚本的真实行为。

### 4.2 当前必须补齐的运行时要素

| 要素 | 目标 |
|---|---|
| 历史逐 bar 执行器 | 稳定驱动历史回放 |
| 时间序列存储 | 支撑 `series[N]`、`bar_index`、历史引用 |
| 变量持久化模式 | 区分 `regular` / `var` / `varip` |
| rollback 引擎 | 实时 tick 重算前恢复 committed state |
| `barstate.*` | 注入 `ishistory` / `isrealtime` / `isnew` / `isconfirmed` / `islast` / `isfirst` |
| Indicator / Strategy 差异 | 默认收盘执行与 `calc_on_every_tick` 区分 |
| Tick 接入与 bar close 检测 | 形成完整实时链路 |

## 6. 当前最高优先级

本轮立刻需要对齐的三项是：

1. **Rollback 引擎**
2. **`barstate.*` 变量**
3. **WebSocket tick 接入 + bar close 检测**

原因：

- 没有 rollback，实时 bar 结果会累计错误，实时指标不可信
- 没有 `barstate.*`，大量脚本无法正确控制 repaint 和收盘确认
- 没有 tick 接入与 bar close 检测，实时计算链路无法真正触发

---

## 7. 本轮执行顺序

为了不把“实时链路”和“执行语义”拆成两条互相等待的线，本轮顺序调整为：

1. **S1-T1：历史 bar 执行器**
2. **S1-T2：时间序列数据结构**
3. **S1-T3：变量持久化模式**
4. **S2-T1：rollback 引擎**
5. **S2-T2：`barstate.*` 注入**
6. **S3-T1 / S3-T2：tick 接入与 bar close 检测**
7. **实时脚本会话协议与前端接入**

原因：

- 先把执行模型底座建稳，再把实时会话接上；
- rollback、`barstate.*` 和 tick 调度属于同一组正确性问题，必须前置；
- 脚本会话和推送协议应建立在正确执行语义之上；
- 前端 JS 指标的统一替换仍留到 WASM 阶段处理。
